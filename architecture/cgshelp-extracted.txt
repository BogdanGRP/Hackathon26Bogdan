CiVision Municipal Service Bus Product
CiVision Municipal Service Bus release 2.7.17 is a product of PinkRoccade Local Government.
Purpose of CiVision Municipal Service Bus
CiVision Municipal Service Bus (CGS) fulfills the role of municipal service bus as defined in the Municipal 
Model Architecture (GEMMA). This basic facility provides the necessary functionality to allow applications
 to exchange data with each other in a responsible manner. These can be data streams, but also 
requests to perform a service. CGS focuses on the technical interfaces, consisting of message structures 
and com-
munication protocols. Where necessary, CGS also provides transformations if the sender and receiver do 
not speak the same 'language'. Applications that communicate with other applications via CGS no longer 
have to implement this themselves.

Purpose of the function
This function is used to trace the communication that takes place via CiVision Municipal Service Bus. 
CiVision Municipal Service Bus is the central collection point for services. Applications can request a 
service (by means of a message). The service request is then processed by the service bus. This 
processing takes place by performing actions.
The following actions (may) apply when handling a service request:
l
Configuration Validation
l
Service license check
l
Authorization
l
Message Validation
l
Routing
l
Storage in Queue (only for asynchronous communication)
l
Transformation (optional)
l
Delivery
l
Request message changed
l
Request message deleted
l
Request message offered again
l
Storage in rejected queue
l
Sending by ebMS adapter (The tooltip of this action shows the progress)
Of all the actions performed, the processing is recorded, depending on the logging settings for the service in
 question (all or only erroneous actions). If an error occurs, the location of the error and the error 
information are recorded and made visible via this screen.
Configuration Validation
Here, the incoming message is tested against the configuration so that the message is syntactically correct and 
an "unknown" message is not simply routed.

Authorization
After the service bus has received a service request, it is first determined whether the requesting party is allowed 
to use the service in question.
Message Validation
The message is validated against the Service Contract. Message validation can consist of validating the 
message against an XSD and/or against programmed validations. Message validation is performed on 
the request message, transformed request message, response message and transformed response 
message. Whether this action is performed depends on the validation settings for the service in 
question.
Routing
A routing is the part of the chain in which the transformation and delivery of a message takes place. 
This routing can vary per message. When communicating via the service bus, a distinction is made 
between synchronous communication and asynchronous communication. In synchronous 
communication, the requesting party waits for a response from the party for whom the message is 
intended. In asynchronous communication, the requesting party does not wait for a response. As 
soon as the message has been successfully offered to the service bus, the communication for the 
requesting party has been completed. It is then the task of the service bus to further deliver the 
message to its destination. In synchronous communication there is always 1 sender and 1 receiver. In
 asynchronous communication, 1 message can be intended for more than 1 recipient. In that case, 
the original message is multiplied and a separate route is determined for each recipient. 
Asynchronous messages are temporarily placed in a message queue. The sending processes in the 
service bus retrieve the "parked messages" from the queue and try to offer them to the recipient. The
 queue always contains the original message. Any transformation is only performed when the 
message is offered to the recipient. This is to prevent the original message from being reconstructed 
due to an error in the transformation.

Storage in Queue
In asynchronous communication, the message is stored in a queue and further processed from there.
Transformation
In message exchange, it is sometimes necessary to convert the message to a different format that is supported 
by the receiving application.
Delivery
Every message has a destination. If the previous actions have all been performed correctly, the message is 
sent to the destination.
Request message modified
If an asynchronous message cannot be processed, the administrator has the option to modify the message 
via the Send Processes function. The message can then be resubmitted to the receiving application. This 
modification action is recorded in the logging.
Request message deleted
If an asynchronous message cannot be processed, the administrator has the option to delete
 the message via the Send Processes function. This deletion action is recorded in the logging.
Request message resubmitted
If an asynchronous message is rejected by the receiving application, the administrator has the option to 
modify the message and resend it via the Send Processes function. This resend action is recorded in the 
logging.
Sending via ebMS adapter
A message that is sent via Digikoppeling enters the
ebMS adapter after leaving the queue for actual sending to the party in question. This is therefore a 
deviation from what is normally the case. The tooltip of this action shows the progress of sending. For 
example: In progress: 3 of 6 attempts to send made

Status
This field can be used to search for the status of the service request.
1.
In progress: the service request is still being processed. This status message will
mainly occur with asynchronous messages. In the case of an asynchronous message,
depending on the so-called Quality of Service, an attempt is made to
offer the message to the application that provides the service a number of times. These messages are first
stored and are periodically offered to the receiving application. If that after
a number of attempts fails, the message can be manually converted by the application administrator to
be converted to status not deliverable.
2.
Invalid: the calling application does not make a correct request. This usually indicates a
application that makes an incorrect request. The service request is rejected "at the gate".
3.
Not deliverable: delivery of the request to the destination is not possible. This
indicates a problem in the adapter (think of an incorrect endpoint, a broken
network connection, the offering application is not available, et cetera).
4.
Processed: all actions have been processed without errors and the message has been delivered.
5.
System error: an action went wrong in the chain. The service request is being processed
taken and an error has occurred.

If the cleanup task is scheduled, it will receive a stop signal at the time specified in this setting. A value of

Digikoppeling what is that?
Like a postman, Digikoppeling does not regulate the content, but the logistics of things such as:
l
What is the address to which the service request must be sent?
l
How does the service provider know who the applicant is?
l
How is the content of a message secured?
l
Has the message been delivered correctly?
In a data flow within the government, the content (the message) is, for example, regulated in Digi- delivery, 
but the packaging (the envelope) is regulated in Digikoppeling. The transport is then regulated by, for 
example, Diginetwerk or goes over the internet.
Digikoppeling is a point-to-point connection between service provider and service applicant. A formal 
communication/service contract is concluded.
To make a Digikoppeling, a software supplier must meet the set of standards. One of the standards is the
 interface, which is WUS (acronym for WSDL, UDDI and SOAP) or ebMS. WUS is for the queries 
(synchronous, best effort request-response) and ebMS for the notifications (asynchronous, reliable). CGS 
naturally supports wsdl and soap.

l
Stable version Digikoppeling WUS 2.0 version 1.2
l

Relevant applications for PRLG in the short term are: BRP (via WUS), NHR (via WUS and for DigiLevering via 
ebMS), LV WOZ (via WUS and ebMS) and BRK (via WUS + large messages).
A relevant version for LV WOZ would be DigiKoppeling version 2.2. Compared to version 
2.1, this version supports large messages. Digikoppeling also knows the Service Register:
The Service Register is a catalog of services that organizations offer via Digikoppeling. It concerns both 
available services and services that are still under development. These services are used within e-
government to exchange data in a standardized manner. About-
Government organizations can publish services in the Register, including all the information that potential 
customers of such a service need to use that service.

The ebMS standard
The ebMS standard is one of the components of the ebXML protocol and stands for ebXML Messaging 
Service. It stems from EDI (Electronic Data Interchange) and is about point-to-point messaging, where 
more is regulated between parties than with wsdl. ebMS is communication protocol neutral, although 
http and smtp are usually the underlying protocols. This is also referred to as a B2B (business-to-
business) connection.
A comparison between wsdl and ebXML can be found here:
l
ws_ebxml_esc.pdf
l
ebxml-ws.pdf
The ebMS standard uses the “Soap with Attachments” standard (SwA) for message transport, a precursor 
to MTOM. Here, soap Header and Body are followed by one or more Mime attachments. Apache Axis2 
supports this out-of-the-box, provided parameters are set to do so, see
.
In addition, both parties must provide a Collaboration Protocol Profile (CPP); for the purchasing
 party (service requester), this means that a so-called consumer specification must be provided.
 This is then combined into a contract called Collaboration Protocol Agreement (CPA). Both are 
xml files. The CPA is then read by adapters from both parties and the connection is established.
Creating a CPA from two CPPs is complex. That is why there are help websites to be found for this:
l
Manual CPA creation
l
open-source-cpa-toolkit
The CPA can indicate which profile is desired; the profile determines whether message signing etc. is 
required.

An important difference from what we have been used to so far is that in ebMS, a simple soap-OK is 
primarily given to an incoming message. After that, an separate acknowledgment is sent 
(asynchronously) via the return channel, whereby the number of attempts etc. is specified in the CPA. 
An ebMS adapter must therefore maintain a relationship between these related - but time-separated - 
message/acknowledgment requests.
This goes so far that entire ebMS conversation scenarios can be agreed between systems in the CPA. This feature 
does not appear to be used in the environments relevant to us.

The flow of an incoming ebMS message
A message from LV WOZ arrives at the ebMS receive process ebMSServlet (httpprocessor) (red
 arrow); this first gives a soap Ack response to LV WOZ. Then the CPA is searched for (green) 
and the message is stored in the MSG Store. After that, an Acknowledge message (yellow 
dotted line with red text) is sent back to LV WOZ (URL is CPA dependent and is taken from it).
Since storage in the MSG Store is done via the MKS CRUD layer, the message can be passed to the 
EBMSINADAPTER via an insert/update event handler. The desired mapping from CPA to provider is defined 
in the settings of this adapter.
This java adapter makes a request and this is placed by CGS Routing Service, depending on the provider set, 
on the correct sending process (Async MSG Queue). If the message is successfully delivered to the sending 
process, it is then marked as processed.
The message is sent by the sending process in the normal way to CBB (read: receiving application) and a Bv03 
response is returned.
The flow of an outgoing ebMS message
An incoming message from CBB (read: requesting application) arrives at the ABAPxxINADAPTER intended 
for LV WOZ (read: receiving application); CBB receives a BV04 response. Message is routed by CGS to the 
EBMSOUTADAPTER; this retrieves the CPA configured for this endpoint (green) and places the message on
 the ebMS service with the EbMSAdapterImpl. sendMessage method.
The message is stored by this service in the MSG Store (as many copies as the number of desired 
sending attempts set in the CPA). Message is extracted from the MSG Store and sent via the ebMS
 Out Sending Threads sending processes (multithreaded, adjustable number) (blue arrow) to LV 
WOZ (URL is CPA dependent and is therefore taken from it). LV WOZ gives a soap Ack response.
An Acknowledge message (yellow dotted line with blue text) then comes (asynchronously) from LV WOZ 
back into the ebMS receive process Servlet (httpprocessor); as soon as this is received, the original (blue)
 message is searched for, the received Acknowledgement is registered with it and the remainder of the 
ready-to-send copies are removed.

Service bus connections and links
Part of product  CiVision Municipal Service Bus .
Purpose of the process
The way applications are connected to the service bus and the way the links from CGS to other 
applications are established determine, among other things, which services are available, which 
applications can use them and via which routes the services run. In order to support a specific chain, a 
distinction is made in:
l
connecting applications (consumers) to CGS to be able to use
certain services. The connection focuses on:
l
setting up certificates
l
authorizing services
l
setting up links to connect CGS to other applications (providers). Here, the
the connection focuses on:
l
setting up adapters
l
testing the connection
An example:
You want to have the possibility to request appointment data from JCC Software from 
CiVision Portal (CPT). In this example, CPT must be connected to CGS as a 'consumer' in 
order to use this facility (JCC service). The correct certificates must be set up correctly for 
CPT and CGS. CPT must also be authorized for the requested service. The connections of 
applications (as consumers) are explained in the various descriptions for connecting 
applications to CGS. It also states which services are in use and includes references to the 
configuration documents of the link from CGS to the other applications (as providers).
Something also needs to be set up on the CGS side; CGS needs to know to which application as 'provider' the
 question from CPT can be asked. In this example, that is the G-BOS module from JCC. For this, it is 
necessary that adapters and adapter endpoints are set up correctly.
The links to providers are explained in the various descriptions for setting up CGS links.
The description below is available for manually setting up new applications as a consumer:
Manually connect applications as consumers.
The description below is available for manually setting up new applications as a provider:
Manually link applications as providers.

Backup and restore tools
Backup and restore tools
Purpose
The backup and restore tools make it possible to export the data from the Makelaarsuite applications and put
 that data back into the database.
Functionality
The Makelaarsuite data is recorded in a database schema per application (Platform, CGS, CML, CMS and 
COO). The Makelaarsuite contains functions to export this data to a file and also to put the data back into 
the database.
The following 2 variants are possible:
A. Complete (full backup and restore)
Intended for 'real' backup and restore where the restore usually takes place in the same set of database 
schemas.
B. Partial (backup data and restore to a different set of database schemas)
The intention here is to only take over the operational data, but not the configuration data. Applicable for 
making a copy from production to test, where the configuration is manually set to test environments and 
test links.
In order not to have to re-arrange the configuration again, in addition to the 2 variants, an additional backup and
 restore function is available for only the configuration data:
C. Partial (backup and restore of configuration data only)
Only the data relating to the configuration is included in the backup.
This is especially useful if the configuration of an already set up test environment needs to be secured.
lig posed. After a full copy from production to test, this secured configuration can be restored. This way, the 
configuration does not have to be set manually every time.
Some comments
Version control: it is ensured that the data model versions of source and target schema are the same. If this is 
not the case, the process is aborted.
With variant A, all data is transferred. When restarting, the relevant MKS instance will send messages
may start sending to the existing url’s / endpoints, as these have not been deleted. So pay attention!
With variant B you can set a number of things using an xml file. This concerns 
Makelaarsuite\MKS##\Database\<application>\ export_import_settings_<application>.xml . There can
the following can be set:
l
Tables that are excluded during export
Name here tables that are not essential to keep, but which (due to their volume) do
significantly increase the required export time
l
Tables that must be (partially) deleted after import (sanitize
For each table, specify a column list with columns to be deleted and default values to be placed in those columns. This can prevent a copy of Makelaarsuite from working
ken with old (incorrect) configuration data.
